7.14 Git Branch

지금까지 master 영역 브랜치

나뭇가지처럼 독립된 개발 환경을 형성 -> 원본(master)에 대해 안전
ex) 로그인/ 게시판 파트마다 따로 개발하는 식

상용 중인 서비스에 난 에러를 해결하려면?
브랜치를 통해 별도의 작업 공간을 만든다. 
master는 최종본만  

Master(main)브랜치의 의미와 역할
기본 브랜치- 저장소의 초기 상태  
타 브랜치 기준점 

git branch 브랜치 목록 확인 
git branch -r 원격 저장소의 브랜치 목록 확인
git branch <브랜치 이름> 새로운 브랜치 생성 <>나[ ] 내의 내용은 변경가능값
git branch -d <브랜치 이름> 브랜치 삭제(병합된 브랜치만 삭체 가능)

git switch
현 브랜치에서 다른 브랜치로 HEAD를 이동시키는 명령어 (우리가 보고있는 위치)
git switch <다른 브랜치 이름> 다른 브랜치로 전환
git switch -c <브랜치 이름> 새 브랜치 생성 후 전환
git switch  -c <브랜치 이름> <commit ID> 특정 커밋에서 새 브랜치 생성 후 전환

HEAD(머리로 생각)

master에서 분기되어 브랜치들 사이서 작업 발생, 병합 등  
master 진행은 다음 업데이트 시만 갱신

git switch 주의사항 -(1/3) 커밋을 기준으로 동작
"git switch 하기 전에, Working Directory 파일이 모두 버전 관리가 되고 있는지 반드시 확인해야 한다."
master <- master
                 ^
              feature 
master 의존성은 과거를 향하기에 실제 진행방향과 반대
버전을 안 찍으면 feature영역서 생성한 게 master에도 존재하게 됨
Git의 브랜치는 독립적인 작업공간을 갖지만 Git이 관리하는 파일 트리에 국한
 
브랜치의 이동은 HEAD가 특정 브랜치를 가리킴
해당 브랜치 최신 commit 을 가리킴

Git Merge 합체

git merge <병합 브랜치 이름>  
대상 브랜치
병합 전 확인 및 주의사항 
1. 수신 브랜치(병합 브랜치를 가져오고자 하는 브랜치) 확인하기
  git branch 명령어를 통해 HEAD가 올바른 수신 브랜치를 가리키는지 확인
  병합 진행 위치는 반드시 수신 브랜치에서 진행

Fast Forward Merge 
브랜치 실제 병합하는 대신 현재 브랜치 상태를 대상 브랜치 상태로 이동시키는 작업(빨리 감기)
Merge과정 없이 단순히 브랜치의 포인터가 앞으로 이동
Master의 헤드가 최신으로 이동할 뿐 - 병합 후 

3-Way Merge 
병합하는 각 브랜치의 commit 2개와 공통 조상 commit 하나를 사용하여 병합하는 작업
새로 커밋 하나가 생긴다 
공통조상(시작점)에서 두 변화량 

VIM에디터->코드 에디터의 조상

VIM 모드 2가지
1. 입력 모드(텍스트 입력)
2. 명령 모드(저장(W), 종료(Q)...)
:wq 누르면 탈출?

Merge Conflict 
병합하려는 두 브랜치가 "동일한 파일의 동일한 부분"에서 변경된 후 병합시 충돌이 발생하는 것
